# Documentation Planning Procedure

Plan pages around reader questions and system concepts, not source files.

Prioritize:

- overview and mental model
- architecture boundaries
- core domain concepts
- important end-to-end workflows
- practical development and operational guides
- API/configuration reference when evidence supports it

For each page define a stable `id`, target `path`, type, audience, purpose, declared evidence/model inputs, required sections, diagram intents, and page dependencies.

Avoid duplicate ownership of the same concept. Prefer links between pages over repeating large explanations.
